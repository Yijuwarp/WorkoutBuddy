# Exercise frequency and difficulty preferences

## Problem Statement

Workouts are currently generated by picking exercises uniformly at random within a Push/Pull/Lower-Body rotation, filtered only by owned equipment. Users have no way to express that they like or dislike a specific exercise, nor any way to control how challenging the exercises they're shown should be. Disliked exercises (or ones that are too hard/easy for the user) appear just as often as any other eligible exercise, with no mechanism to bias selection toward favorites or away from exercises the user wants to avoid.

## Solution

Add two new preference dimensions that bias workout generation:

1. **Per-exercise frequency** — Always / Often / Less / Never, settable on any individual exercise, controlling how likely (or whether) it's selected.
2. **Difficulty ceiling** — Easy / Medium / Hard, a single setting that filters the eligible exercise pool to exercises at or below the chosen ceiling.

New users are asked to set a difficulty ceiling the first time they reach `WorkoutScreen`, before their first workout is generated, followed by a one-time "setting up your workout" animation. Per-exercise frequency tagging is exposed afterward, in-context on exercise cards and via a new "Manage Exercises" screen, rather than during onboarding.

## User Stories

1. As a new user, I want to pick a difficulty ceiling before my first workout, so that my very first session is appropriately challenging.
2. As a user who dislikes a specific exercise, I want to mark it "Never," so that I stop seeing it in my workouts.
3. As a user who loves a specific exercise, I want to mark it "Always," so that it reliably shows up when relevant.
4. As a user with a mild preference, I want "Often" and "Less" options, so that I can nudge selection without fully forcing or excluding an exercise.
5. As a user, I want my difficulty ceiling to apply to every exercise I could be shown, including the cardio exercise appended to each workout, so that the setting is consistent everywhere.
6. As a user, I want the equipment I own to still be respected even for exercises I've marked "Always," so that I'm never shown an exercise I physically can't do.
7. As a user, I want to tag exercise frequency right from the workout screen when I see an exercise I have a strong opinion about, so that I don't have to leave my workout to do it.
8. As a user, I want a dedicated screen listing all exercises, so that I can review and adjust my frequency preferences for exercises I haven't encountered yet in a workout.
9. As a user who hasn't tagged a given exercise, I want it treated neutrally, so that untagged exercises behave exactly as they do today.
10. As a user opening the app for the very first time, I want a short "setting up your workout" moment after choosing my difficulty, so that the app feels like it's doing something purposeful before showing me a workout.

## Implementation Decisions

- **Difficulty model:** Add a fixed, hand-assigned `difficulty` field (EASY / MEDIUM / HARD) to `ExerciseEntity`, analogous to existing fields like `category` and `impactLevel`. Not computed from strength/stamina scores.
- **Difficulty selection is a ceiling, not multi-select:** The user picks one tier (Easy, Medium, or Hard) via a single control. Selecting a tier includes that tier and everything below it (e.g. "Hard" → Easy + Medium + Hard all eligible). Stored as `difficultyCeiling` on `UserProfileEntity`.
- **Difficulty filtering is a hard gate**, applied identically to the equipment-ownership filter: exercises above the ceiling are excluded from the eligible pool before any random selection happens. Applies to every selection path, including the auto-appended cardio exercise.
- **Per-exercise frequency model:** New `ExercisePreferenceEntity` (exerciseId, frequency) stores only exercises the user has explicitly tagged. No row = untagged = neutral (equivalent to today's uniform-random weight, not a synonym for "Often").
- **Frequency tiers and their effect on selection:**
  - **Always** — guaranteed inclusion. Inserted into the day's workout unconditionally (subject only to the equipment and difficulty hard gates below), and explicitly **overrides** the existing diversity cap (max 2 exercises per body part per workout). Remaining slots are filled by the normal random/weighted pick around the guaranteed exercises.
  - **Often** — selection-weight multiplier of approximately 1.5x neutral in the random pick (doesn't need to be exact).
  - **Less** — selection-weight multiplier of approximately 0.5x neutral in the random pick (doesn't need to be exact).
  - **Never** — hard exclude from the eligible pool, same mechanism as the equipment and difficulty filters.
  - Applies uniformly to the cardio pool as well as the main Push/Pull/Lower-Body pool.
- **Filter precedence:** Equipment ownership and difficulty ceiling are hard gates that nothing overrides, including "Always." "Always" only overrides the diversity cap.
- **Empty-pool fallback:** If "Never" exclusions (combined with equipment/difficulty filters) leave too few eligible exercises to fill a category or the cardio slot, generate a shorter workout for that category rather than relaxing the "Never" exclusion or the equipment/difficulty gates.
- **Onboarding flow:** The existing 7-step `OnboardingScreen` is unchanged and remains profile-basics only. There is no dedicated difficulty-ceiling picker step or "setting up your workout" animation — onboarding doesn't ask about difficulty at all. Instead, `difficultyCeiling` is proxied from the gym-experience level the user already picks during onboarding (Beginner→EASY, Intermediate→MEDIUM, Expert→HARD) and set at profile creation, so new users start at a sensible ceiling without an extra decision. It can be changed anytime afterward from Profile settings (a slider control between Easy and Hard).
- **Post-onboarding frequency tagging — two entry points:**
  1. A quick action (e.g. overflow menu) on each exercise card in `WorkoutScreen`, to tag frequency in-context as exercises are encountered.
  2. A new "Manage Exercises" screen, reachable from `ProfileScreen`, listing the entire exercise library with a frequency selector per exercise. Exercises requiring equipment the user doesn't currently own are still shown and still taggable (visually marked, e.g. greyed out), so preferences are preserved if the user adds that equipment later.
  - Changes made through either entry point apply silently/instantly.
- **Persistence:** New Room entity `ExercisePreferenceEntity`; new `difficulty` column on `ExerciseEntity`; new `difficultyCeiling` column on `UserProfileEntity`. Requires a real, explicit `Migration` (not the existing destructive-fallback wipe-and-reseed pattern) for this version bump, which adds the new columns, backfills `difficulty` per exercise, and sets `difficultyCeiling = 'MEDIUM'` for existing profiles — gym experience (which new profiles proxy the ceiling from) isn't a reliable signal to backfill for pre-existing profiles, so MEDIUM is used as a neutral guess rather than guessing EASY or HARD. `fallbackToDestructiveMigration()` remains in place as the catch-all for any other/future version gap. Existing users can adjust the ceiling anytime from Profile settings.
- **Difficulty seeding:** Initial `difficulty` values are derived automatically from each exercise's existing `impactLevel` (HIGH→HARD, MEDIUM→MEDIUM, LOW→EASY) as a starting point, both in the migration backfill and in seed data for new exercises, then hand-tuned afterward for outliers rather than rated from scratch.
- **"Always" vs. the per-workout exercise count:** Guaranteed inclusion wins over the normal per-workout exercise count. If a category has more "Always"-tagged eligible exercises than the usual slot count (e.g. 4), all of them appear; the count acts as a floor for the remaining weighted/random filler picks, not a hard ceiling on total exercises.

## Testing Decisions

- Workout generation logic changes (difficulty hard-gate, frequency weighting, "Always" diversity-cap override, "Never" hard-exclude, empty-pool shorter-workout fallback) are core business logic changes and should be covered by unit tests on `WorkoutViewModel`'s generation function, including:
  - An "Always" exercise appears even when it would normally be blocked by the diversity cap.
  - An "Always" exercise is still excluded if it requires unowned equipment or exceeds the difficulty ceiling.
  - A "Never" exercise never appears in generated output.
  - A category with too many "Never" exclusions produces a shorter workout instead of an error or a relaxed exclusion.
  - The cardio pick respects difficulty ceiling and frequency tags the same way the main pool does.
- The first-launch overlay gating logic (`difficultyCeiling == null` → show overlay, else skip) should have a test or at least a manual check against existing-user migration behavior.
- The "setting up your workout" animation itself, like other animation/motion work in this codebase, is verified manually by running the app per repo convention, not via automated assertion on timing/visuals.
- Existing tests that exercise workout generation (if any assert on selection counts/composition) should be reviewed for compatibility with the new filtering steps.

## Out of Scope

- Per-exercise frequency tagging during first-launch onboarding (deferred to post-onboarding, in-context and Manage Exercises screen).
- Difficulty computed dynamically from strength/stamina scores — difficulty is a fixed, hand-assigned exercise attribute in this pass.
- Replaying the "setting up your workout" animation on later preference changes — it is strictly a first-launch, one-time moment.
- Filtering the "Manage Exercises" list down to only equipment-compatible exercises — the full library is always shown.
- Any change to the equipment-ownership filter's precedence — it remains a hard gate, unaffected by frequency or difficulty settings.
